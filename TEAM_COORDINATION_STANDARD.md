# Team Coordination Standard

Status: permanent engineering governance, migrated from issue #40
("Define Team Jumpstile issue-based coordination protocol"), which served
its purpose defining this protocol but is not the right long-term home for
it (see `CONSTITUTION.md`, "Long-lived policy does not live in an issue
tracker"). Issue #40 now points here instead of containing the policy
itself.

Governed by `CONSTITUTION.md` (top) and subject to `PROJECT_IDENTITY_STANDARD.md`
for anything touching identity, branding, or attribution -- both take
precedence over anything in this document if a conflict is found.

---

## 1. Purpose

Use GitHub issues as the shared coordination layer for Team Jumpstile work
across human and AI contributors.

Goals:
- Keep all contributors aware of current work.
- Make issues the durable source of truth for task status and findings.
- Ensure investigations produce actionable outputs, not buried comments.
- Prevent duplicate work, stale assumptions, and hidden decisions.
- Prevent automated back-and-forth loops between contributors.

## 2. Compliance requirement

Every comment or update posted under this protocol must comply with
`PROJECT_IDENTITY_STANDARD.md`, without exception:

- No self-attribution footer of any kind (e.g. "Generated by \[tool
  name\]"), regardless of what a given tool's default output template
  normally includes. Strip it before posting.
- No AI tool/vendor name anywhere in the comment body -- not in the text,
  not in a footer, not in a link. Use "primary reviewer" / "secondary
  reviewer" per the roles defined below, never a specific product name.
- No personal name other than a real GitHub username that a human
  contributor actually posted under (a tester's own handle in their own
  comment is fine; a coordinator referring to that person by real first
  name is not -- use their GitHub handle or a role like "tester").
- This applies retroactively: a past comment found violating the above is
  corrected the next time that issue is touched, not left standing.

## 3. Required issue update format

When posting an investigation or handoff update, use a concise structure:

```text
Status:
Finding:
Evidence:
Confidence:
Remaining unknowns:
Next action:
Needs response from:
Loop guard:
```

## 4. Loop guard rule

Every coordination update must say whether it needs a response.

Use one of these values:

- `Needs response from: Primary reviewer`
- `Needs response from: Secondary reviewer`
- `Needs response from: Jumpstile`
- `Needs response from: nobody`

If no response is needed, say so explicitly.

The primary reviewer and the secondary reviewer must not respond to each
other just because the issue changed. They should respond only when the
update asks for their role specifically, or when a new finding materially
changes the task.

## 5. No ping-pong rule

Do not post a new issue update that only restates agreement.

Acceptable updates: new evidence, corrected finding, implementation result,
test result, security finding, architecture decision, blocked status,
request for a specific review, final disposition.

Not acceptable: "Agreed," "Sounds good," repeating the prior comment,
replying only because another contributor replied.

## 6. Durable artifact rule

Important findings must not stay only in issue comments.

- Regression scenario -> Pester test (or equivalent), tracking issue if
  deferred, and a quality-gates document until automated.
- Vulnerability -> fix or security issue.
- False positive -> `LESSONS_LEARNED.md`.
- Accepted risk -> `SECURITY.md` or an ADR.
- Architecture discovery -> code comment, ADR, and/or `LESSONS_LEARNED.md`.

## 7. Handoff rule

Every meaningful handoff should answer: what changed, why it changed, what
evidence supports it, what remains unknown, what should happen next, and
who should act next.

## 8. Primary reviewer instructions

The primary reviewer should continue monitoring issues, but must apply the
loop guard before responding.

If an issue update says `Needs response from: nobody`, the primary
reviewer should not post a response unless it finds a materially new
problem.

If an issue update asks for secondary-reviewer input, the primary reviewer
should not answer on the secondary reviewer's behalf. It may continue
implementation work only if the next action clearly belongs to the
primary reviewer.

If an issue update asks for primary-reviewer investigation, it should
investigate and post findings in the required format.

## 9. Acceptance criteria

- Team Jumpstile uses GitHub issues as the coordination hub.
- Issue updates are concise and actionable.
- The primary reviewer and the secondary reviewer do not create update
  loops.
- Important findings become tests, docs, ADRs, security notes, quality
  gates, or tracked issues.
- The repository remains the source of truth, not private chat history.
- No coordination update violates `PROJECT_IDENTITY_STANDARD.md` (Section 2
  above); a violation found after the fact is corrected, and if it recurs
  from the same source, the root configuration producing it is flagged
  directly rather than only patching the individual comment.
