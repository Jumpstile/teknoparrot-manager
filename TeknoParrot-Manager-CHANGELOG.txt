-------------------------------------------------------------------------------
  v0.99.11 BETA  (2026-06-21)
-------------------------------------------------------------------------------

  Fix: the Eggman collection dat parser silently dropped roughly half of
  all games from its lookup index, regardless of whether they were
  otherwise fully supported. The parser skips over each game's (often
  hundreds of) checksum entries for performance; after skipping, it was
  also advancing to the next entry a second time, and once that drift
  happened to land exactly on a game's closing tag, that whole game was
  never indexed at all. Confirmed against a real collection dat: of 493
  games, only 236 were being indexed before this fix; 476 are indexed
  after it. This was the real cause behind several previously-reported
  "game not recognized" cases (including OutRun 2 and Time Crisis 5,
  both of which have perfectly valid catalog entries) being unable to
  auto-register even though nothing was actually wrong with their entry
  or profile.

-------------------------------------------------------------------------------
  v0.99.10 BETA  (2026-06-21)
-------------------------------------------------------------------------------

  Fix: control propagation's Input API correction (added in v0.99.2 for
  "button"-family games whose archetype uses MergedInput) only ever ran
  as part of the same pass that copies button bindings. A game that was
  already fully bound from an earlier run -- including every game bound
  before v0.99.2 shipped -- is treated as "already configured, nothing
  to do" and skips that whole pass, so it could never have its Input API
  retroactively corrected even after upgrading. Propagation now checks
  and fixes the Input API on an already-bound profile too, without
  touching any of its button bindings. (issue #1)

-------------------------------------------------------------------------------
  v0.99.9 BETA  (2026-06-21)
-------------------------------------------------------------------------------

  Fix: a two-executable game (e.g. Initial D: Arcade Stage Zero, which
  needs a second GamePath2 alongside the regular GamePath) only had that
  second path field set when the script created the UserProfile for the
  very first time. Any profile that already existed from an earlier run
  -- including every profile registered before this GamePath2 feature
  itself was added in v0.99.6 -- was treated as "already registered,
  nothing to do" and skipped right past the GamePath2 check, leaving it
  permanently blank. Registration now also backfills GamePath2 on an
  already-registered profile if it's missing and the companion
  executable is found, instead of only setting it during a brand-new
  registration. (issue #8)

-------------------------------------------------------------------------------
  v0.99.8 BETA  (2026-06-21)
-------------------------------------------------------------------------------

  Fix: a game folder could still be wrongly listed as needing manual
  registration even though its real executable was already correctly
  registered, if that same folder also contained a second, unrelated
  exe-like file (a generic/shared launcher stub) whose name happened to
  collide with other profiles. The list of games still needing
  registration never excluded a folder once a DIFFERENT file in that
  same folder had already matched cleanly elsewhere in the same run --
  only the separate "completely unrecognized" list did that check.
  Confirmed against a real tester log: Nosferatu Lilinor's folder
  contains both its real launcher (NLAM.exe, already registered under
  the correct profile) and a generic "main" stub that collides with two
  unrelated racing-game profiles -- the stub alone was enough to put the
  already-fully-configured game back on the action-required list every
  run. (issue #9)

-------------------------------------------------------------------------------
  v0.99.7 BETA  (2026-06-21)
-------------------------------------------------------------------------------

  Fix: hardened the registration check that's supposed to recognize a
  game whose UserProfile is already correctly configured, even when the
  folder's executable name is shared by several profiles. The check
  compares the profile's saved game path against the file actually found
  on disk, but only stripped a trailing backslash from the saved value,
  not other incidental whitespace -- so a saved path with any extra
  whitespace would never match and the game would be wrongly flagged as
  still needing registration. (issue #9)

-------------------------------------------------------------------------------
  v0.99.6 BETA  (2026-06-21)
-------------------------------------------------------------------------------

  Fix: profiles that require two executables (HasTwoExecutables=true --
  e.g. Initial D: Arcade Stage Zero/Ver. 2/The Arcade and its companion
  amdaemon.exe process) only ever had their primary GamePath configured
  during registration. The second executable's required GamePath2 field
  was never set, leaving these profiles half-configured even though
  registration reported success. Registration now also looks for the
  profile's ExecutableName2 alongside the already-resolved primary exe
  and sets GamePath2 when found, for all of its registration paths.
  Never overwrites an existing GamePath2 and never blocks registration
  of the primary exe if the companion file isn't found. (issue #8)

-------------------------------------------------------------------------------
  v0.99.5 BETA  (2026-06-21)
-------------------------------------------------------------------------------

  Fix: Write-DownloadAudit (the SHA256 logging done for every third-party
  binary this script downloads -- the Eggman dat ZIP, the BepInEx release
  ZIP, and the FFBArcadePlugin DLLs) silently swallowed a missing or
  locked source file instead of logging it as a hashing failure.
  Get-FileHash raises a non-terminating error by default, so the
  existing try/catch never actually caught it -- the audit log ended up
  with a malformed entry showing an empty hash instead of the intended
  "could not hash" message. No security impact (this logging never
  gated or blocked anything), but the audit trail itself was misleading
  on failure. Fixed by adding -ErrorAction Stop so the catch behaves as
  documented.

-------------------------------------------------------------------------------
  v0.99.4 BETA  (2026-06-21)
-------------------------------------------------------------------------------

  Fix: registration could mark an entire game folder as "accounted for"
  the moment any exe-like file inside it (an installer, uninstaller,
  redistributable stub, or other non-launcher file) happened to share a
  filename with an unrelated profile's executable name -- even when that
  collision itself only produced an unresolved "needs manual
  registration" result. This permanently blocked the dat-lookup and
  profile-code fuzzy-match passes from ever getting a chance to resolve
  the folder's real launcher via a different exe in the same folder.
  Registration now only marks a folder accounted for when an exe's own
  outcome was conclusive (registered, already registered, or a genuine
  profile-code conflict) -- a "shared, below threshold, no dat match"
  result no longer blocks the later passes from still trying. Suspected
  root cause for several "folder exists, profile exists, never
  connected" reports (issue #10).

-------------------------------------------------------------------------------
  v0.99.3 BETA  (2026-06-21)
-------------------------------------------------------------------------------

  Fix: a folder whose game was already correctly registered in
  TeknoParrotUI could still be wrongly flagged as needing manual
  registration, when that folder's executable name is shared by several
  TeknoParrot profiles (e.g. a generic Lindbergh/Sega ALLS loader name).
  The existing-registration check only looked at the profile codes the
  script itself considered as fuzzy-match candidates for the folder
  name -- if the real, already-configured profile's code didn't fuzzy-
  match the folder name closely enough (or wasn't a candidate at all,
  because its own template uses a different executable name), the
  folder was flagged anyway even though nothing needed to be done.
  Registration now also checks, before any fuzzy-name matching, whether
  any existing UserProfile already points its GamePath at this exact
  executable -- a name-independent signal that catches both cases.
  Confirmed against two real tester examples (issue #9).

-------------------------------------------------------------------------------
  v0.99.2 BETA  (2026-06-21)
-------------------------------------------------------------------------------

  Fix: on newer TeknoParrot builds, FFB Blaster setup (mode 7) and the
  Library health check's FFB Blaster coverage report could find zero
  eligible games even with a paid membership. The identifying text moved
  from the field's name to its category label in newer GameProfile XML
  (CategoryName "FFB Blaster" with a generic FieldName of "Enable",
  instead of the field name itself containing "blaster"). Discovery now
  checks the category label first, falling back to the old field-name
  check for older profiles. Confirmed against real GameProfile XML from
  a tester's clean TeknoParrot install (issue #2).

  Fix: control propagation (mode 6) could copy a real, correct button
  binding into a target profile but leave its buttons appearing empty in
  the TeknoParrot UI. The propagated archetype's Input API (e.g.
  MergedInput) was being silently rejected when the target profile's
  on-disk FieldOptions hadn't listed it yet -- TeknoParrot's own UI
  dynamically adds that option to a profile's XML the first time it is
  selected there, so an absent entry doesn't mean the profile can't use
  it. Set-ProfileInputApi now mirrors that behavior for MergedInput
  specifically; other API values still require the option to already be
  listed. Confirmed against tester diagnostics across several rounds
  (issue #1).

  v0.99.1 BETA  (2026-06-20)
-------------------------------------------------------------------------------

  Fix: the third-party FFB plugin setup (mode 7) could crash with an
  unhandled "Value cannot be null" error right after FFB Blaster setup
  aborted (e.g. when no FFB Blaster field was found, or no membership).
  Root cause was a PowerShell pipeline quirk where an empty-array return
  value gets silently turned into $null by the time a plain variable
  assignment captures it -- fixed at every affected return point, plus a
  defensive null-guard added at the actual crash site.

  Fix: the exact same quirk meant pressing "A) All unextracted games" in
  the AutoSync game picker silently extracted nothing instead of
  extracting everything. Fixed the same way.

  Fix: Remove-PostgresPartialInstall (used before a fresh PostgreSQL
  install attempt) now verifies a matched registry entry's install
  location before ever uninstalling it, and only ever stops/removes a
  service with the exact expected name -- closes a theoretical gap where
  an unrelated standalone PostgreSQL 8.3 install (for other dev work)
  could have been caught by the cleanup. Found via external security
  review; no evidence this was ever actually triggered.

  v0.99 BETA  (2026-06-19)
-------------------------------------------------------------------------------

  New: Postgres setup (mode 11). Installs and configures the local
  PostgreSQL 8.3 database that some Incredible Technologies games need
  (Golden Tee Live, Power Putt Live, Silver Strike Bowling Live, Target
  Toss Pro, Orange County Choppers Pinball). Detects which of your
  registered games need it automatically -- no manual list to maintain.
  If PostgreSQL is already installed, it is never reinstalled or
  modified; if a game's database already exists, it is never recreated
  or restored over. For games whose profile predates TeknoParrot's own
  "Automatically create Database" feature, this mode creates the
  database and restores that game's bundled backup itself; for newer
  profiles, it just fills in the connection settings and lets TeknoParrot
  handle the rest on first launch. The database password is encrypted
  (Windows DPAPI) before being saved -- the first secret this script has
  ever stored.

  New: Postgres database backups. Every time mode 11 runs, it backs up
  every existing Postgres database first, before touching anything.
  Restore them via menu 9 (Restore backup), alongside the existing
  TeknoParrot UserProfiles and LaunchBox library backup choices.

  Note: installing PostgreSQL requires running this script as
  Administrator (the first feature in this script that does). Setting up
  connection fields and backing up/restoring databases for an
  already-installed PostgreSQL does not require elevation.

  v0.98 BETA  (2026-06-19)
-------------------------------------------------------------------------------

  New: direct LaunchBox integration. After a run, the script can now write
  your registered games straight into LaunchBox's own library files --
  no import wizard needed. Checks LaunchBox/BigBox are closed first, backs
  up every file it is about to change into Scripts\LaunchBoxBackups\, and
  never duplicates a game already present. The first time you use it, you
  choose how TeknoParrot games should appear: mixed into your existing
  Arcade platform, a separate "TeknoParrot" platform, a platform with a
  name you choose, or both Arcade and a dedicated platform at once. The
  choice is remembered for next time. New games have no box art/metadata
  yet -- use LaunchBox's own Search per game to fetch it, same as for any
  manually-imported game.

  New: a "LaunchBox library backup" option under menu 9 (Restore backup)
  restores the exact LaunchBox files the direct integration last changed,
  the same way the existing UserProfiles restore already worked.

  Fix: the LaunchBox manual-import reference file (TeknoParrot-LaunchBox-
  Import.xml, still available if you prefer not to let the script touch
  LaunchBox directly) had its ApplicationPath field pointing at the game's
  executable. Confirmed against a real working LaunchBox installation that
  this is wrong -- LaunchBox's own TeknoParrot emulator entry launches by
  profile (--profile=%romfile%.xml), so the path it actually needs is the
  GameProfile XML itself. Also fixed the printed import-wizard instructions,
  which told users to import *.exe files from the staging folder; the
  correct target is the UserProfiles\*.xml profile files.

  Internal: extracted the seven near-duplicate config-save blocks scattered
  across the script into a single Save-Config function, since adding new
  settings (the four LaunchBox fields above) to seven copies by hand would
  have been a real risk of missing one.

  v0.91 BETA  (2026-06-18)
-------------------------------------------------------------------------------

  Security hardening -- found and fixed four real issues affecting how
  untrusted, live-fetched filenames were used in path construction. None
  were known to be exploited; this closes the gaps proactively.

  - Path traversal (Medium): Invoke-FFBPluginSetup deployed a DLL using a
    destination filename parsed from the live-fetched AutoSetup.cmd file
    (untrusted, fetched from GitHub every run) without checking it stayed
    inside the game's own folder. A crafted rename line (e.g. containing
    "..\..\") could have written outside the intended folder. Fixed with
    a containment check before the file copy.
  - Path traversal (Medium): Invoke-BepInExUpdateCheck built its download
    path from the GitHub Releases API's asset filename (also untrusted)
    without sanitizing it the same way. Fixed by stripping to a bare
    filename and verifying containment before downloading.
  - Path traversal (Low-Medium): the Eggman dat downloader's default save
    path had the same gap (asset filename from the GitHub API, used
    unsanitized as the default path offered to the user). Fixed the same
    way; if the filename is ever unsafe, the script now falls back to its
    existing manual-path-entry flow instead of using an unsafe default.
  - Long-path bug (Low): Expand-ZipFileSafe's "\\?\" long-path prefix
    (used to bypass the 260-character MAX_PATH limit during extraction)
    was malformed for UNC destination paths (\\server\share\...) -- only
    matters if a staging or game folder is a literal UNC path rather than
    a mapped drive letter. Fixed with the correct "\\?\UNC\server\share\..."
    form for that case; drive-letter paths (the common case) are unaffected.

  All four fixes verified with synthetic tests (malicious filename input
  confirmed blocked; normal/legitimate input confirmed still works).

  v0.88 BETA  (2026-06-18)
-------------------------------------------------------------------------------

  New feature: BepInEx update check (mode 8).

  - BepInEx is a third-party Unity plugin/modding framework some games
    need (Family Guy Bowling, Mars Sortie, NERF Arcade, Rainbow
    BomberGirl, Super Bikes 3, TMNT, and others). This checks every
    registered game that already has BepInEx installed against the
    latest stable release on GitHub and offers a single batched update.
  - Never installs BepInEx into a game that doesn't already have it.
    Only the latest STABLE 64-bit build is ever used -- never 32-bit,
    never a pre-release. A 32-bit install is left alone and reported
    separately, never silently touched or downloaded for.
  - Existing BepInEx files are backed up (to BepInEx_Backup_<timestamp>
    inside the game's own folder) before any update is applied.

  Baked into the script: two automatic compatibility warnings that used
  to live only in a separate, soon-to-be-unavailable game notes
  document. Both run automatically on every AutoSync/Register run as
  new ACTION REQUIRED items, with step-by-step fix instructions -- no
  separate mode needed:

  - Raw Thrills path-length limits: specific older Raw Thrills titles
    have hard-coded limits on how long their full install path can be
    (roughly 64 or 96 characters depending on engine era). Exceeding it
    means the game won't launch. The script now checks the known
    affected titles automatically and tells you the exact short folder
    name the community has settled on for each one.
  - iDmacDrv32.dll version pins: five BlazBlue-series games
    (BBCF, BBCP, BlazBlue Continuum Shift, Continuum Shift II, Cross Tag
    Battle) need a SPECIFIC OLDER version of iDmacDrv32.dll -- a newer
    version causes a coin error. The script checks the file's CRC32
    against the known-required value and flags a mismatch with the
    required CRC32 and where to get the correct file. Never auto-fixed
    (the usual "let TeknoParrot redeploy its current copy" fix would
    make this specific case worse, not better).

  Fix: README.md (the GitHub homepage) was missing the entire Force
  Feedback section added in v0.87 -- it only had a mode-table row. Also
  fixed a factual error in the staging-folder rules (network drives ARE
  supported, an older sentence said they were not) and several more
  stale mode-number references left over from the v0.87 menu reorder.

  v0.87 BETA  (2026-06-17)
-------------------------------------------------------------------------------

  New feature: Force feedback (FFB) setup (mode 7).

  - Two independent mechanisms, both optional, neither requires the other:
    - FFB Blaster: TeknoParrot's own built-in force feedback. Paywalled --
      requires an active, paid TeknoParrot membership or it has no effect.
      The script asks plainly before touching anything; the field name is
      discovered dynamically by scanning GameProfiles at runtime, never
      hardcoded. Your UserProfiles are backed up first, same as every
      other destructive operation in this script.
    - Third-party plugin (mightymikem/FFBArcadePlugin): free, no
      subscription needed. The per-game destination-DLL table and the DLLs
      themselves are fetched live from GitHub every run -- nothing is
      bundled or hardcoded, so newly supported games are picked up
      automatically with no script update.
    - Real overlap between the two is substantial -- roughly half the
      third-party table also has a native FFB Blaster field, confirmed
      by cross-referencing the live data, not assumed. Too many for a
      per-game prompt, so the script resolves matches first, then asks
      ONE question covering every overlapping game at once: keep native
      for all of them, or use the third-party plugin for all of them.
      Never silently defaults to either side.
    - If the third-party plugin's destination DLL filename is already
      occupied (e.g. ReShade using d3d9.dll for the same game), that game
      is skipped with a warning -- never overwritten.

  Menu reorder: modes renumbered so related features sit together --
  AutoSync/Register first, then Crosshair/ReShade/dgVoodoo2/GPU fix/FFB as
  a group, then Restore backup/Library health check, then Exit. See the
  updated MODES section in the README for the new numbering.

  Bug fix: the Eggman dat collection's source repository
  (Eggmansworld/Datfiles) posted its final release and moved to
  Eggmansworld/TeknoParrot, which uses date-based release tags instead of
  a fixed tag. Get-EggmanDatRelease now queries that repo's "latest"
  release instead of a fixed tag -- dat downloads would otherwise have
  silently stopped working once the old repo's last release aged out.

  v0.86 BETA  (2026-06-17)
-------------------------------------------------------------------------------

  Bug fix (a second issue found from the same real run as v0.85, reported
  as "X-Games Snowboarder and Winter X Games didn't show as manual or
  missed -- they just weren't anywhere"):

  - For shared-executable games (Raw Thrills' sdaemon.exe and similar),
    when a folder's name didn't fuzzy-match any candidate well, the script
    checked "is ANY candidate sharing this exe already registered?" -- and
    if so (e.g. H2Overdrive already had a profile), it skipped the
    dat-lookup/manual-registration fallback ENTIRELY for that folder, even
    though the folder's own game (a different title sharing the same
    generic loader) was never actually registered. The folder vanished
    completely: not registered, not flagged ambiguous, not even logged.
    Fixed: a sibling being already registered no longer short-circuits the
    fallback. Only the folder's own best-guess candidate specifically
    being already registered counts as "this folder is handled" -- so an
    unrelated sibling's registration no longer hides a folder that still
    needs attention.
    Verified with a control test against the pre-fix code (reproduced the
    exact disappearance) and a fix-confirmation test, plus re-ran the
    v0.85 false-positive and genuine-duplicate regression scenarios to
    confirm this change doesn't reintroduce either of those.

    v0.80 fix. For shared-executable games (NESiCAxLive's game.exe and
    similar), a folder whose own name does not fuzzy-match any candidate
    well falls back to an "is any candidate already registered?" check
    that scans the WHOLE shared candidate pool -- and marks every match it
    finds as claimed, including titles that have nothing to do with that
    folder. If one of those incidentally-marked titles' OWN folder is
    processed later in the same run and resolves to its own code with high
    confidence, the old check order treated "this code was already marked
    seen" as an automatic conflict, before ever checking whether the code
    was simply already correctly registered. The result: dozens of
    perfectly fine, already-registered games (Aquapazza, Crimzon Clover,
    Ikaruga, and ~130 others in one real run) were reported as needing
    manual re-registration in TeknoParrotUI for no reason.
    Fix: Register-Games now snapshots which profile codes already existed
    BEFORE the run started. A code being "seen" only counts as a genuine
    conflict if it was claimed by ANOTHER folder DURING this same run;
    if it was already registered beforehand (by TeknoParrotUI or an
    earlier run), it's correctly reported as "Already set" instead.
    Verified with two tests: one reproducing the exact false-positive
    against the pre-fix code as a control (confirmed it failed the same
    way the real run did), and one confirming genuine same-run conflicts
    (e.g. multiple Virtua Fighter 5 ROM revisions sharing one profile,
    the original v0.80 scenario) are still correctly surfaced, not
    silently swallowed by this fix.

  Four follow-on enhancements:

  - dgVoodoo2 per-game config override: dgVoodoo2Presets\<ProfileCode>.conf
    overrides the global dgVoodoo.conf for one game, same convention as
    ReShadePresets. Unlike the global conf (which never overwrites an
    existing destination file), a per-game override always overwrites --
    it's an explicit per-game choice, so it must take effect even on a
    game that already has a conf deployed from a prior run.
  - Structured duplicate-profile-code resolution: right after registration,
    if any "Reason=duplicate" conflicts were found (one profile code
    claimed by two folders), the script now offers to resolve them on the
    spot -- shows which folder currently holds the profile and which folder
    is contesting it, then lets you [K]eep, [S]witch, or [Q]uit. Resolved
    entries no longer appear in ACTION REQUIRED. "Shared" conflicts (one
    exe name mapping to several different games) are unaffected -- those
    still require manual registration in TeknoParrotUI since there's
    genuine ambiguity about which profile applies, not just which folder.
  - New mode 8: Library health check. Read-only -- reports how many
    registered profiles have a valid, broken, or empty GamePath, lists the
    affected codes, and shows the last full run's summary line from the
    log. No extraction, registration, repair, propagation, or network
    access. Mode 8 (Exit) is now mode 9.
  - Invoke-GpuFixSetup's WMI GPU-vendor query now has a 10-second
    -OperationTimeoutSec bound, so a WMI service hiccup falls through to
    the existing manual-vendor-entry prompt instead of hanging. Lower
    urgency than the v0.83 Test-IsNetworkPath fix since local GPU
    enumeration doesn't have the network-round-trip hang risk a mapped
    network drive does, but kept consistent defensively.

    (> 9 digits = skip) is now applied before every [int] cast.

  - International title matching: Get-NormalizedGameKey used [^a-zA-Z0-9]
    which stripped all non-ASCII letters, collapsing e.g. "Pokémon" to
    "pokmon" and erasing Japanese characters entirely. Changed to [^\p{L}0-9]
    so Unicode letters are preserved and Dice scores remain valid.

  - XML comment bypass in Get-PrimaryExecutableName: the fast regex path ran
    against raw XML text, so a commented-out <ExecutableName> node earlier in
    the file would be matched instead of the real one. XML comments are now
    stripped ((?s)<!--.*?-->) before the regex runs.

  - Array append overhead in Expand-NumberList: $out += $n rebuilt the entire
    array on every iteration. Switched to List[int] with .Add() for O(1)
    appends; converted back to array at return.

  - README fix: HyperSpin 2 troubleshooting section incorrectly referred to
    an ".xml ROM entry" (HyperSpin 1 terminology). Updated to describe the
    actual HyperSpin 2 JSON game-list file.

  v0.38 BETA  (2026-06-11)
-------------------------------------------------------------------------------

  Bug fixes / security:

  - Expand-ZipFileSafe: GetFullPath was called on the combined base+entry path
    before the \\?\ prefix was applied. On .NET 4.x (PS 5.1 runtime),
    GetFullPath calls the Win32 API without extended-length-path support and
    throws PathTooLongException for any path over 260 chars -- meaning the
    function failed at exactly the case it was written to solve. Fixed by
    switching to a component-level zip-slip guard (regex check for .. and
    rooted paths) that never calls GetFullPath on a long string.

  - Expand-ZipFileSafe: zip-slip guard used StartsWith without a trailing
    backslash. A ZIP entry resolving to C:\games\FooBar\evil.exe would pass a
    guard anchored on C:\games\Foo because the string is a prefix match. Fixed
    by using the component-level guard above, which is not vulnerable to this
    prefix-collision attack.

  - Expand-ZipFileSafe: removed redundant Directory.Exists check before
    CreateDirectory. CreateDirectory is idempotent and does not throw when the
    directory already exists. The check introduced a TOCTOU race window (another
    process could delete the directory between Exists and CreateDirectory).

  - AutoSync: partial extraction folder was not cleaned up when
    Expand-ZipFileSafe threw mid-extraction (e.g., disk full, I/O error). On
    the next run, the sentinel file is gone and the folder has some content, so
    the code incorrectly classified the game as already extracted and marked it
    up-to-date in the sync state -- permanently skipping re-extraction of a
    corrupt game. The inner catch now removes the partial folder before
    recording the failure.

  - All New-Item -ItemType Directory calls replaced with
    [System.IO.Directory]::CreateDirectory(). PowerShell's New-Item -Path
    interprets [ ] * ? as wildcard characters, so a path like
    E:\Games\TeknoParrot [Staged] could resolve to the wrong location or fail
    silently. The .NET method has no wildcard interpretation, is idempotent,
    and works in PS 5.1 without requiring -LiteralPath.

  v0.34 BETA  (2026-06-10)
-------------------------------------------------------------------------------

  Bug fixes:

  - Get-NormalizedGameKey: replaced lazy-quantifier bracket regex `\[.*?\]`
    with character-class form `\[[^\]]*\]`. Both behave identically for
    well-formed names, but the new form is explicit that the match must not
    cross a `]` boundary, which is more correct for malformed names that
    contain an unmatched `[`.

  - Get-GameFiles: forward slashes in the caller-supplied $folder path (or in
    a file's FullName) caused Split('\') to produce an incorrect base depth,
    which could wrongly include or exclude extension-less Linux binaries.
    Both the base path and each FullName are now normalised to backslashes
    before the depth calculation.

  - Select-GamesInteractive: added a defensive guard so that a
    `$start > $end` paging edge case (e.g. after a future refactor) returns
    an empty page rather than throwing a PowerShell slice error.

  - Measure-PathThroughput: reduced the read-sample cap from 100 MB to 20 MB
    to avoid large heap allocations on systems with constrained memory or very
    large ZIPs. A 20 MB read is more than sufficient for an accurate MB/s
    estimate.

  Bug fixes:

  - Export-HyperSpinJson: roms detection was not PS 5.1-safe. PS 5.1's

  v0.26 BETA  (2026-06-10)

  Initial release.

    TeknoParrot GameProfiles templates, copies them to UserProfiles with the

  - Fuzzy name matching. Games that share a single executable (NESiCAxLive,

    all unextracted games. Paginated list with queue tracking.

  - Control propagation. Bind one game of each type; the script copies those
    controls to every other game of the same type by function match. Carries
    aim-mode settings (relative input, sensitivity, hide-cursor) between
    same-type games.

  - Device survey. Asks which controls you have and prints a tailored plan
    of which game to bind with which device.

  - Game repair. Finds broken or empty GamePaths and re-points them to the
    correct executable when the match is unambiguous.
